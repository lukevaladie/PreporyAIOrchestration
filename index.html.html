§2 · Data pipeline

How the sources connect to the surfaces.

Each source has a small connector that turns what it produces — a doc edit, a meeting transcript, a deal update — into events on a shared stream. The hub reads from that stream and updates its graph. Everything the user sees (brief, query, meeting commits) reads from the graph.

Adapters
One per source. Webhook-first, polling where the source doesn't push. Knows nothing about downstream.
Event stream
Shared append-only log. Deterministic IDs — idempotent re-ingest. The integration seam.
Event store
Replayable. Graph is derived state; change the derivation, replay the store to rebuild.
Hub
World model + alignment graph. Typed nodes (KR, task, doc, meeting, decision, person) and typed edges, derived on write.
Capabilities
Readers over the hub state. Commit extractor is the only one that writes back — closing the loop.
§3 · Monday morning

What it looks like when you log on.

Overnight, adapters ingest and the brief generator composes. By 8 a.m., the layer has drafted proposed next steps for each priority — outlines, messages, decision options — and surfaced adjacent work from other orgs. The user's job is taste: approve, edit, or redirect.

§4 · Cross-org collaboration

Cross-org collaboration surfaces.

The links that lead from a KR to a design decision about Rory also connect a person at Prepory to a person at Deans of Admissions working on a similar problem. For anyone whose account spans orgs, this could be a central use case of the orchestration layer.

Adjacent work
Surfacing parallel efforts across orgs.
Luke is refining Rory's feedback on essay concluding paragraphs at Prepory. Sara Fernandes is building coach curriculum on the same topic at Deans of Admissions. The brief surfaces Sara's curriculum drafts and logged decisions to Luke, and flags Luke's open questions to Sara.
Shared infrastructure
Workflows built once, reused across orgs.
N2Growth built an automated research workflow for executive-search briefs and thought-leadership pieces — pulling sources, summarizing, drafting. Prepory could use the same pattern for college-research briefs. Different content, same pipeline. The graph surfaces the existing workflow before Prepory builds a parallel one.
Capacity
Matching open time to stuck work.
A Prepory coach has a light week coming up. Someone at N2Growth is stuck on something they could help with. The hub shows the overlap to both sides — either can pick it up or pass.
Expertise
Locating the person who's done it before.
Asking "who's shipped this before" returns a person and the artifact that proves it. Crosses org boundaries by default for users whose accounts span orgs; stays in-org for everyone else.
§5 · Open questions

What's worth deciding together.

Blocker

Is Google Chat in v1?

Vault export (Enterprise tier, batch) vs opt-in bot in key spaces (real-time, partial) vs defer. This changes whether the pipeline is real-time end-to-end, which in turn changes what the brief can show.

Philosophy

What counts as a "connection" worth surfacing?

Sharing the same KR tag is easy but noisy. Semantic adjacency is powerful but opaque. If people stop trusting the people-connection surface, the whole cross-org premise falls apart — so the bar has to be high from day one.

Philosophy

What crosses an org boundary, and who chooses?

Decisions, yes. KR titles, probably. Open questions, maybe. Raw documents, probably not. The default matters more than the override — it sets whether cross-org becomes the norm or the exception.

Philosophy

How do we handle the layer being wrong?

The brief will mis-flag drift. The router will suggest bad matches. Trust erodes from false positives faster than it compounds from correct hits. The dispute primitive is a product question, not an ML one.

Philosophy

What does this displace, and for whom?

If the brief replaces the routing-context work middle management currently does, the org has to decide what those humans do instead. Naming it here because it should shape the product — not be handled after the fact.
